### 方案 4: L1 MoveIt 规划 (4 次失败 + 1 次成功)
- 目标: 调 /plan_kinematic_path service 验证 OMPL 能规划 0 → q_pick
- **失败 1 — IK 输出超 URDF 限位**:
  - 现象: `Joint joint_2 is constrained to be above the maximum bounds. Assuming maximum bounds instead`
  - 根因: numerical_ik_simple 算出 q_pick[1] = 1.81 rad, 超 joint_2 上限 1.5708 rad 达 0.24 rad
  - 修法: 加 JOINT_LIMITS 常量 + clamp_to_joint_limits(q) (5-14)
  - 结果: ❌ 仍 FAIL — 限位修了但 OMPL 仍拒绝
- **失败 2 — launch_ros 嵌套结构 normalize 失败**:
  - 现象: "Failed to normalize given item of type '<class 'float'>'"
  - 根因: ompl_planning.yaml 加 default_workspace_bounds (list-of-floats) + planner_configs (nested dict), yaml.safe_load → parameters=[dict] 路径不接受
  - 修法: 试了 ParameterFile (move_group 解析 ROS2 格式失败), 试了内联 dict (仍 fail)
  - 结果: ❌ 撤回 — 走回标量 ompl_planning.yaml
- **失败 3 — is_diff=True/False 切换**:
  - 现象: "Detected jump back in time. Clearing TF buffer" + "Skipping invalid start state"
  - 归因: TF jump back 引起 current_state_monitor 失效 (5-16)
  - 结果: ❌ 仍 FAIL
- **失败 4 — RViz GUI 手动 Plan**:
  - 现象: GUI 点击 Plan 也失败, "Catastrophic failure"
  - 根因: 不是 headless 限制, 是真问题
  - 结果: ❌ 仍 FAIL — 排除了 headless 假设
- **成功 — /check_state_validity service 诊断**:
  - 工具: rclpy + GetStateValidity service, 5 行 rclpy
  - 输出: `valid=False, contacts=[link_6 ↔ tool0 depth=-0.0003]`
  - 根因: **URDF 中 tool0 是 2cm 球, 中心在 link_6 末端, 与 link_6.stl 几何有 0.3mm 物理重叠**. MoveIt state validity 永远 False
  - 修法: SRDF 加 1 行 `<disable_collisions link1="link_6" link2="tool0" reason="Adjacent"/>`
  - 结果: ✅ L1 规划 status=OK, err=1, fraction=1.0, n_points=7

## 根本原因分析

1. **L1 失败 4 次都归因错** (TF sync / yaml 嵌套 / workspace bounds / IK 限位) — **事实不驱动**, 是经验主义. 之前没调过 `/check_state_validity` 直接验
2. **OMPL 错误信息不具体**: "Skipping invalid start state" 没说为什么 invalid — 必须用 state validity service 拿 contacts
3. **CLAUDE.md 5-10 "P5 baseline 闭环 PASS" 实际只验了 L3→L4**: L1 自始至终没通过. CLAUDE.md 5-13 "用 RViz 手动 Plan" 也只是绕开, 没真修
4. **SRDF 是 MoveIt 行为的"宪法"**: 任何 URDF 加新 link 必同步检查 SRDF disable_collisions, 我**没做**这个 cross-check
5. **诊断工具 vs 改配置** — 我反复改 yaml (4 次), 没用 5 行诊断服务. **效率差 100 倍**

## 关键教训

1. **OMPL "Skipping invalid start state" 第一步**: 调 `/check_state_validity` 看 contacts. 不要改 yaml 嵌套, 不要加 workspace bounds, 不要怀疑 TF sync. **5 行代码定位 1 行修复**
2. **SRDF 是宪法**: URDF 加 link (特别是末端工具 link) 必同步检查 SRDF collision matrix 是否有 disable. tool0 是 2cm 球, 中心在 link_6 末端, 跟 link_6.stl 几何重叠是已知设计, 必须标 Adjacent
3. **CLAUDE.md 5-17 必看**: 未来 P5 baseline L1 失败, 先看这条 — 用 `/check_state_validity` 5 行诊断, 不要原地打转改 yaml
4. **同一 bug 也影响 P5 baseline (gofa.srdf)**: 顺手补了, 避免下次 P5 demo.launch.py L1 也炸
5. **GUI 探针 A 路径价值**: 排除了 "headless 限制" 假设, 确认是真配置问题. 10 分钟探针换 1 小时原地打转, 划算
6. **P8 反思**: 嘴上说"事实驱动", 实际**没调诊断服务就开始改 yaml** — 是 3.25 行为. 把 `/check_state_validity` 写进 5-17 强制流程, 才是这次失败最大的复利
